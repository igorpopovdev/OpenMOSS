# Проект задачи: Ежедневное обновление китайских технологических новостей на английском сайте

> Этот документ — полный проект для долгосрочной циклической задачи. Планировщик использует его как справочник при создании задач.
> Тип задачи: `recurring` (долгосрочный цикл)

---

## 1. Цель задачи

Управление английским технологическим новостным сайтом **Example Site** (`https://www.example.com`), ориентированным на глобальную аудиторию и освещающим последние события в области **AI, технологий, гаджетов и автомобилей** в Китае. Каждый рабочий цикл производит高质量ные английские статьи — от сбора материалов до публикации в виде стандартизированного конвейера.

## 3. Красные линии безопасности контента

> ⚠️ Эти правила применяются ко всем этапам (сбор, написание, публикация, рецензия). При обнаружении нарушения — немедленно исключить.

**Запрещено:**

- 🚫 **Политическая чувствительность** — политические позиции, споры о политиках, международные конфликты, идеологическая конфронтация
- 🚫 **Насилие и кровь** — описания насилия, террористических актов, кровавых сцен
- 🚫 **Порнография и пошлость** — любое涉及色情 содержание
- 🚫 **Дискриминация и ненависть** — расовая, гендерная, региональная дискриминация, религиозная ненависть
- 🚫 **Дезинформация** — непроверенные слухи, теории заговора, псевдонаука
- 🚫 **Нарушение авторских прав** — прямое копирование原文 (цитирование с указанием источника и переписывание своими словами разрешено)
- 🚫 **Финансовые рекомендации** — рекомендации по акциям, криптовалютам
- 🚫 **Утечка приватности** — публикация личных данных без согласия

## 5. Определения подзадач

### ① AI Сяоу — Сбор новостей

**Обязанности**: Собирать самые горячие AI/техно/гаджеты/авто новости за день из китайского интернета

**Результат**:

- Стандартизированный отчёт о сборе (Markdown), включающий:
  - 📌 Top 3 обязательных тем дня (с полным原文 содержанием)
  - 📝 3–5 рекомендуемых тем
  - 📎 Дополнительные материалы
  - 🔮 Наблюдения о трендах

**Критерии приёмки**:

1. Отчёт покрывает минимум 2 области (AI/техно/гаджеты/авто)
2. Каждая обязательная тема содержит полное原文 содержание (не резюме)
3. Каждая тема с ссылкой на источник, индикатором популярности, обоснованием выбора
4. Актуальность в пределах 48 часов
5. Каждая новость верифицирована минимум 2 источниками
6. Не содержит нарушений красных линий безопасности

---

### ② AI Цзянгуа — Создание контента

**Зависимость**: Подзадача ① (отчёт AI Сяоу)

**Обязанности**: На основе отчёта создавать статьи на китайском и английском

**Результат**:

- Каждая статья содержит версию на китайском + английском (Markdown)
- Разные типы по важности материалов:
  - 📰 Быстрая заметка (300–500 слов): оперативная информация
  - 📝 Глубокая статья (1500–3000 слов): детальный анализ + мнение
- Каждая статья содержит frontmatter: title, title_en, category, keywords, summary, summary_en

**Критерии приёмки**:

1. Каждая статья на обоих языках
2. English — это локализованная адаптация, не машинный перевод
3. Заголовок содержит ключевые слова, SEO-дружелюбен
4. Резюме ≤ 60 символов
5. Факты точны, соответствуют отчёту, без выдуманных данных
6. Корректная категоризация (быстрая заметка / глубокая)
7. Китайские специфические концепции снабжены пояснениями в английской версии
8. Не содержит нарушений красных линий безопасности

---

### ③ AI Сяокэ — Публикация WordPress

**Зависимость**: Подзадача ② (статьи AI Цзянгуа)

**Обязанности**: Публикация статей в WordPress + SEO/GEO оптимизация

**Результат**:

- Английские статьи опубликованы в WordPress
- Лог публикации (ID статей, ссылки, категории, теги)

**Критерии приёмки**:

1. English-версия опубликована, категории и теги корректны
2. Title Tag ≤ 60 символов, с ключевыми словами
3. Meta Description ≤ 160 символов
4. URL Slug короткий, с ключевыми словами, на английском
5. Иерархия заголовков корректна (H1 единственный, H2/H3 упорядочены)
6. Минимум 2 внутренних ссылки (на связанные статьи)
7. Статус статьи: сначала draft, после проверки — publish
8. Опубликованный контент не содержит нарушений красных линий

## 6. Механизм рецензии (⭐ Ключевой этап)

### Роль рецензента: Синсин

Синсин — **страж качества** всего конвейера. Не получает задачи от Планировщика, а активируется по таймеру и主动но сканирует подзадачи в статусе рецензии (`st list --status review`).

### Процесс рецензии

```
Исполнитель st submit → подзадача переходит в review
                        ↓
Синсин активируется → st list --status review → находит задачи
                        ↓
st get <id> смотрит критерии + log list --action delivery
                        ↓
Проверяет файлы в рабочей директории
                        ↓
         ┌─── Соответствует ──→ review create <id> approved <4–5 баллов>
         │              → подзадача done ✅
         │              → исполнитель получает баллы
         │
Сверяем ─┤
         │
         └─── Не соответствует ─→ review create <id> rejected <1–2 балла> --issues "конкретные проблемы"
                        → подзадача rework 🔄
                        → с исполнителя списываются баллы
                        → при следующем пробуждении исполнитель видит rework, читает issues и исправляет
```

### Чеклист рецензии

#### Рецензия отчёта о сборе (Сяоу)

| Пункт           | Стандарт                                    |
| ----------------| ------------------------------------------- |
| Покрытие областей | Минимум 2 области                          |
| Полнота контента | Обязательные темы содержат полное原文       |
| Ссылки на источники | Каждая тема с ссылкой                      |
| Актуальность     | В пределах 48 часов                         |
| Кросс-проверка   | Каждая новость минимум 2 источника           |
| Красные линии    | Без политики/насилия/порно                  |

#### Рецензия статей (Цзянгуа)

| Пункт          | Стандарт                                  |
| ---------------| ----------------------------------------- |
| Оба языка      | Китайский и английский оба есть            |
| Качество перевода | English — локализация, не машинный перевод |
| SEO заголовок  | Ключевые слова, привлекает клики          |
| Длина резюме   | ≤ 60 символов                             |
| Факты          | Соответствуют отчёту, без выдумок          |
| Формат         | frontmatter полный, категория корректна     |

| SEO элементы  | Title Tag ≤ 60, Meta ≤ 160                 |
| URL Slug      | Короткий, английский, с ключевыми словами   |
| Иерархия H   | H1 единственный, H2/H3 упорядочены         |
| Внутренние ссылки | Минимум 2 связанные статьи             |
| Красные линии | Без нарушений                             |

### Критерии оценки

| Балл | Значение                   | Действие                      |
| ---- | -------------------------- | ----------------------------- |
| 5    | Превышает ожидания          | Принято, +5 баллов исполнителю |
| 4    | Полностью соответствует      | Принято, +5 баллов            |
| 3    | Соответствует с мелкими огрехами | Принято, ±0            |
| 2    | Частично не соответствует    | Отклонено, −5 баллов         |
| 1    | Серьёзно не соответствует / нарушение красных линий | Отклонено, −5 баллов |

### Требования к отклонению

- `--issues` обязательно, чётко описывает конкретные проблемы
- Формат: «Проблема 1: xxx; Проблема 2: xxx»
- При нарушении красных линий указать тип нарушения
- После рецензии отправить в канал уведомлений: «Результат: название | исполнитель | ⭐x/5 | комментарий»

## 7. Циклический механизм

Задача имеет `type: recurring`. После завершения цикла Планировщик создаёт подзадачи следующего раунда:

- Подзадача ① (сбор) done → подзадача ② (написание) может начинаться
- Подзадача ② (написание) done → подзадача ③ (публикация) может начинаться
- Подзадача ③ (публикация) done + рецензия прошла → цикл окончен
- При следующем пробуждении Планировщик видит все done → создаёт новые подзадачи того же названия

> ⚠️ Подзадачи имеют зависимости: ① → ② → ③. Последующие подзадачи читают результаты предыдущих. Планировщик должен указать зависимости и пути к файлам в описании подзадачи.

## 10. Инструкции для Планировщика

При создании задачи разбивайте так:

```bash
# 1. Создать родительскую задачу
task create "Tech News Daily" --desc "См. prompts-private/task-daily-news.md" --type recurring

# 2. Активировать
task status <task_id> active

# 3. Создать подзадачи первого раунда (соблюдать порядок зависимостей)
st create <task_id> "Сбор новостей" \
  --deliverable "Стандартизированный отчёт (с полным原文)" \
  --acceptance "2+ области; полное原文; источники и популярность; в пределах 48ч; без нарушений" \
  --assign <xiaou_agent_id> --type recurring

st create <task_id> "Создание контента" \
  --deliverable "Статьи на китайском и английском (Markdown)" \
  --acceptance "Оба языка; локализация; SEO заголовок; ≤60 символов резюме; факты точны; без нарушений" \
  --assign <jianggua_agent_id> --type recurring

st create <task_id> "Публикация" \
  --deliverable "Лог публикации WordPress" \
  --acceptance "English опубликована; SEO оптимизирован; категории/теги корректны; ≥2 внутр. ссылки; без нарушений" \
  --assign <xiaoke_agent_id> --type recurring
```

> Рецензент (Синсин) не получает задач — активируется по таймеру и сам сканирует `st list --status review`.
> В следующем цикле Планировщик просто создаёт новые подзадачи того же названия; исполнители автоматически продолжат работу.
